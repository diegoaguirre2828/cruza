# Tracking Audit — Cruzar

**Date:** 2026-04-26
**Method:** ripgrep scan + file inspection (lib/trackEvent.ts, lib/trackShare.ts, lib/trackClick.ts, lib/useSessionPing.ts, /api/events/track, /api/user/touch)
**SDK:** No third-party product analytics SDK installed. In-house wrapper writes to Vercel Analytics + Supabase `app_events` table.

## Summary

- **81 `trackEvent()` call sites across 37 files**
- **53 distinct event names**
- **0 ORPHANED events** — every event name found in code is also called from at least one site
- **0 `identify()` calls, 0 `group()` calls** — no user identity or account binding for product analytics
- **Identity is implicit:** the server route `/api/events/track` calls `supabase.auth.getUser()` at write time and stamps `app_events.user_id`. No user traits ever flow to a destination, only the FK.
- **Naming is consistent:** snake_case across all sites
- **Centralization is partial:** all calls go through one wrapper (`lib/trackEvent.ts`), but event names are string literals inlined at each call site (no central enum)
- **Three sister wrappers** with their own server endpoints: `trackShare` (`/api/user/share-event`), `trackBusinessClick` + `trackLinkClick` (`/api/track/click`), `useSessionPing` (`/api/user/touch`, heartbeat — not events)

## Share + Click Sub-Systems (separate from trackEvent)

**Share events** — `trackShare(channel, context)` -> `POST /api/user/share-event`. Increments user `share_count` server-side. Used by 7 components. Channels observed: `copy`, `whatsapp`, `facebook`, `native`, `native_success`.

**Business clicks** — `trackBusinessClick(...)` -> `POST /api/track/click`. Used in `app/negocios/page.tsx` for phone/whatsapp/website outbound clicks on business directory listings.

**Link clicks** — `trackLinkClick(...)` exported from `lib/trackClick.ts` but no callers found in current scan.

**Session presence** — `useSessionPing()` -> `POST /api/user/touch`. Not an event tracker — heartbeats `last_seen_at` + device traits onto `profiles` row. Only fires for signed-in users.

## Identity Management

- **identify():** none called
- **group():** none called
- **page() / screen():** none called explicitly; Vercel Analytics auto-tracks page views via `<Analytics />` component in `app/layout.tsx`
- **Logout/reset:** no analytics-side reset observed; Supabase auth handles session.
- **Session id:** client-generated `cruzar_session_id_v1` in `sessionStorage` (base36 timestamp + random); included on every event payload sent to `/api/events/track`.
- **Traits-on-profile** (NOT analytics traits, just DB columns): `last_seen_at`, `first_seen_at`, `last_seen_device`, `last_seen_os`, `last_seen_browser`, `install_state` (web | pwa | twa | capacitor) — written by `/api/user/touch`.

## Observed Patterns

| Pattern | Observation |
|---------|-------------|
| Naming style | snake_case, consistent |
| Centralization | Single wrapper `lib/trackEvent.ts`. Sister wrappers `trackShare`, `trackBusinessClick`, `trackLinkClick`. |
| Event-name source | String literals inlined at each call site — no central event-name enum or constant file |
| Type safety | `eventName: string`; `props: Record<string, string \| number \| boolean \| null \| undefined>` — no per-event type contract |
| Server-side write | `/api/events/track` resolves auth user, rate-limits 120 events/hour per user/IP, caps event_name at 60 chars and props at 2000 bytes |
| Failure mode | Fire-and-forget, never throws, never blocks user |
| `keepalive: true` on fetch | yes — events survive page navigation |
| Tier in properties | Inconsistent — `one_tap_alert_*` carry `tier`, most events don't |
| Account/group dimension | Never present in any property — Business-tier events do not carry account context |
| Property-shape consistency | Loose — `report_submitted` and `affiliate_clicked` have varying keys across call sites |

## Hygiene Observations

- **Duplicate-name with varying shapes:** `report_submitted` (4 sites) and `affiliate_clicked` (5 sites) pass overlapping but not identical properties.
- **`pwa_grant_pending`** fires from two components (`PWASetup.tsx` and `ClaimProInPwa.tsx`).
- **Two CTA event families for camaras** (`camaras_install_cta_*` from `CamarasStickyInstallCta.tsx`, `camaras_sticky_*` from `StickyCamarasCta.tsx`) — overlapping intent.
- **`servicios_page_view`** is a manually fired page event; rest of the app relies on Vercel Analytics auto page-views. Cross-system inconsistency.
- **Verbosity bias:** install/PWA/IAB CTAs have 30+ events between them — heavy on impression/dismiss/choice tracking. Most other product surfaces (favorites, smart-route, leaderboard, business dashboard) have zero events.

## Notes for the design phase

- All call sites already route through `lib/trackEvent.ts`. Adding a third sink (PostHog) means editing one file — keep this wrapper, do not rewrite the 37 callers.
- Event names are stable and `snake_case` — usable as PostHog event names without renaming.
- Identity binding (`identify` / `group`) is the biggest structural addition. Currently the server stamps `user_id` on `app_events`, but no user traits ever leave the database.
- PostHog supports nested groups natively, so the Business-account hierarchy (account -> drivers / shipments) can be modelled with `group("business_account", id, traits)` on dispatcher actions.
